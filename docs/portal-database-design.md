# قاعدة بيانات الموقع (PortalDb) — PostgreSQL

قاعدة بيانات **منفصلة تماماً** عن:
- قاعدة الأمين (SQL Server / `mt000`, `cu000`, …)
- قاعدة إدارة الـ API (`ApiManagementDb` — مستخدمون وصلاحيات الـ API)

تعمل على **السيرفر الخارجي** مع موقع **PHP**. لا ترحيل من ملفات XML القديمة.

## المعمارية

```text
[زائر / عميل ويب / موظّف موقع] → PHP → PostgreSQL (portal_db)
                                      ↓
                            api_proxy + JWT (حساب خدمة ApiManagementDb)
                                      ↓
                            API محلي → SQL Server (أمين)
```

### الصور

عند الرفع من لوحة الموقع:
1. حفظ الملف على السيرفر الخارجي (عرض سريع).
2. إرسال نفس الملف إلى API الأمين (`/api/material-images/…`).
3. تسجيل الربط في `portal_images` (مسار محلي + `amine_image_guid`).

### الأسعار وسياسات الوصول

- **الأسعار** تُجلب دائماً من API الأمين (مصدر واحد).
- **الاختلاف** بين زائر / رابط مشاركة / عميل مسجّل عبر **`access_policies`** (إظهار سعر، سلة، طلب، فلاتر…).

### قنوات العرض

| القناة | الجدول / الإعداد |
|--------|------------------|
| الصفحة الرئيسية | `home_sections` + فلاتر أو منتجات يدوية |
| متجر عام | `store_guest_settings` → `access_policy_id` |
| رابط مشاركة | `share_links` + `share_link_filters` |
| عميل مسجّل | `web_customers.access_policy_id` (بعد الموافقة) |

## مستخدمو الموقع (موظفون)

جدول `web_users` — **ليس** `ApiUsers`.

أدوار مقترحة (تُبذر في `web_roles`):
- `super_admin` — كل الصلاحيات
- `sales` — روابط، طلبات، زوّار
- `content` — أقسام الرئيسية، إعدادات الشركة
- `customers_admin` — موافقة/تفعيل عملاء الويب ومنح السياسات

## عملاء الويب (منفصلون عن cu000)

### التسجيل (الخيار 1 + 2 مع موافقة)

| المصدر | الحالة الابتدائية | بعد موافقة الأدمن |
|--------|-------------------|-------------------|
| `self_register` | `pending` | `active` + `access_policy_id` |
| `admin_created` | `active` (افتراضي) أو `pending` حسب اختيار الموظّف | نفس المنطق |

حالات `web_customers.status`:
- `pending` — بانتظار الموافقة (لا دخول للمتجر المخصّص)
- `active` — مفعّل
- `rejected` — مرفوض
- `suspended` — موقوف

حقول الموافقة: `approved_by_web_user_id`, `approved_at`, `rejection_reason`.

### نفس الأسعار، سياسات مختلفة

عند التفعيل يُعيَّن `access_policy_id` (مثلاً: يرى كل الفئات + يطلب، أو عرض فقط…).

## الطلبات وربط الأمين (لاحقاً)

الجدول `orders` يحتوي حقولاً جاهزة:
- `amine_bill_guid`
- `amine_sync_status` (`none` | `pending` | `synced` | `failed`)
- `amine_synced_at`, `amine_sync_error`

لا يُنفَّذ الربط في المرحلة الأولى.

## حساب خدمة API

مستخدم في `ApiManagementDb` (مثل `portal-service`) بصلاحيات قراءة مواد وصور (+ رفع صور حسب الحاجة).
بيانات الاعتماد في `.env` على سيرفر PHP فقط — لا تُرفع إلى Git.

## اللغة

واجهة لوحة الموظّفين: **عربي فقط**. أسماء الحقول في قاعدة البيانات عربية/تقنية حسب الحاجة (`title` بدون إلزام `title_en`).

## الملفات

- `docs/portal-db-schema.sql` — DDL كامل PostgreSQL
- `docs/portal-db-seed.sql` — أدوار وصلاحيات وسياسات افتراضية

## ترتيب التنفيذ

1. تنفيذ DDL + seed على PostgreSQL الخارجي
2. إنشاء حساب خدمة في ApiManagementDb
3. تحديث `api_proxy.php` (JWT + مسارات API الجديدة)
4. طبقة PDO في PHP واستبدال XML
5. الصفحة الرئيسية + لوحة التحكم
6. موافقة العملاء + متجر عام + روابط
7. رفع صور مزدوج
8. (لاحقاً) مزامنة الطلب → فاتورة/حجز أمين
